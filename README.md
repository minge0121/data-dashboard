# 数据大屏的自适应实现？
## 关键：
    不能像普通网页那样"布局跟着变"，因为图表位置、大小比例是精确排版好的。不能让布局结构跟着屏幕变（把三列变两列），这会影响画面的信息密度和视觉平衡。
## 方案选择：
-  1.不选rem/vw单位。虽然普通网页最佳实践是rem/vw单位，实现页面的等比例缩放。但Echarts内部有大量px，无法全部覆盖故而会导致图表变形。
-    2.不选@media查询断点写不同样式。因为这会改布局，而且大屏需要调的数值太多，难以维护。
-   3.选择scale 方案：以设计稿为基准，计算缩放比，整体缩放。即根据屏幕宽度和高度，计算出一个缩放比例，将所有元素的宽度和高度都乘以这个比例，实现等比例缩放。比如：按 1920×1080 写代码，写完之后整体缩小/放大塞进你的屏幕里
### scale 缩放后 ECharts 图表会模糊吗？
    不会。ECharts 图表在缩放后，会自动调整像素密度，保持清晰度。
### 性能问题：
    CSS transform: scale() 的缩放发生在浏览器的合成器层，是 GPU 加速，不触发重排和重绘，故不会影响性能。
## 总结：
    代码永远基于设计稿尺寸（如 1920×1080）
           ↓
    useScreenScale 算出 scale 和 offset
           ↓
    CSS transform: scale() 整体缩放
           ↓
    浏览器 GPU 加速渲染，自动处理高清屏
           ↓
    结果：任何屏幕上看到的画面，和设计稿一模一样，只是大小不同


## 开发问题：
1.底部两个模块（产业结构占比、实时事件监控）图表在页面偏下方，内容显示不全
问题出在主内容区采用了 3 列布局，但中列（6/12宽度）只有一个图表，而左右列（各3/12宽度）各有两个图表，导致左右列的垂直空间被压缩，特别是底部两个模块（产业结构占比、实时事件监控）显示不全
优化方案：
- 改用 CSS Grid 显式行布局 替代之前的 Flex 嵌套，彻底避免 flex 高度计算问题
-   grid-rows-2：强制将主区域分为上下两行，每行占 50% 高度
-   grid-area 精确指定每个模块的位置，不依赖 DOM 顺序
